The rover navigation subsystem executes on the NVIDIA Orin NX 16GB under ROS~2 Humble and uses the Nav2 stack for autonomous path planning, trajectory tracking, and obstacle avoidance. After a goal pose or waypoint sequence is issued, the rover performs autonomous motion without continuous operator input.

Navigation is implemented as a closed-loop pipeline:
(1) sensor acquisition,
(2) costmap update,
(3) global path planning,
(4) local trajectory generation, and
(5) velocity command execution.
This loop runs continuously, allowing the rover to react online to terrain changes and newly detected obstacles.

Global planning is performed by NavFn using Dijkstra's algorithm over a 2D costmap with \qty{0.05}{\metre} resolution, producing a discrete path as \texttt{nav\_msgs/Path}. Local motion is generated by the DWB controller, which samples candidate trajectories in velocity space $(v,\omega)$ within a dynamic window defined by the current velocity and acceleration limits. Candidate trajectories are scored according to path tracking, obstacle clearance, and goal alignment; only dynamically feasible trajectories that avoid impassable cells are considered. The selected command is published as \texttt{geometry\_msgs/Twist}.

Traversal is executed as autonomous waypoint following:
(1) receive a goal pose or next waypoint,
(2) compute a global path,
(3) track it using DWB,
(4) update the costmap from live sensor data,
(5) perform local trajectory adjustment if the path is obstructed,
(6) trigger global replanning if local recovery fails, and
(7) stop once the goal tolerance is satisfied.
The same navigation stack is reused in other mission tasks: during traversal, the rover follows waypoint sequences autonomously; during sampling and maintenance, it autonomously positions itself at task-specific goal poses, while fine interaction is delegated to dedicated subsystems (\autoref{par:arm_kinematics_motion_planning}).

The navigation costmap is populated by the sensor suite described in \autoref{par:navigation_sensor_suite}. Stereo depth from the front camera and range data from the three ToF sensors are integrated as 3D occupancy observations in the voxel layer. LiDAR scans are inserted into the obstacle layer as 2D obstacle observations. These data sources are fused into a layered costmap used by both the global planner and the local controller.

The layered costmap consists of:
the \textit{voxel layer}, which maintains a 3D occupancy representation at \qty{0.05}{\metre} resolution from stereo and ToF data;
the \textit{obstacle layer}, which projects relevant 3D structure into the 2D planning map and incorporates LiDAR detections;
and the \textit{inflation layer}, which applies a \qty{0.15}{\metre} safety margin around obstacles based on the rover footprint (\qtyproduct{0.60 x 0.55}{\metre}).

For navigation, an obstacle is any region that is non-traversable for the rover footprint. This includes occupied cells from LiDAR or depth sensing, terrain protrusions above \qty{0.15}{\metre}, depressions deeper than \qty{0.10}{\metre}, and all cells with cost $\geq 253$ after inflation.

Terrain classification augments the costmap with traversability estimates. Traversable terrain is assigned costs in the range $0$--$252$ based on slope and roughness, while cells with cost $\geq 253$ are treated as impassable. Traversable terrain variations remain admissible for planning and do not require avoidance, whereas impassable obstacles require local trajectory adjustment or global replanning.

Collision avoidance is implemented at two levels. At the planning level, NavFn computes paths only through traversable cells. At the control level, DWB rejects all local trajectories intersecting cells with cost $\geq 253$. Obstacle handling therefore follows a two-stage strategy: traversable terrain is crossed without unnecessary deviation, while impassable obstacles trigger local trajectory adjustment or global replanning to maintain collision-free motion.

If the current path becomes obstructed, the navigation state machine shown in Figure~\ref{fig:behavior_sm} executes:
(1) a local evasion attempt,
(2) global replanning if local recovery fails and an alternative path exists, and
(3) fail-safe stop if no safe trajectory can be generated within a defined timeout.
